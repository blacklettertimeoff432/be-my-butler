# BMB v0.4.0 Design Document

**Date**: 2026-03-19
**Status**: Approved
**Scope**: 5 features — OMX cross-model, Superpowers discipline, session-end prep, parallel sessions, visual brainstorming

## Overview

BMB v0.4.0 is a major upgrade with five independent features that collectively address cross-model reliability, agent quality, workflow continuity, parallelism, and user experience.

**Implementation Order** (by dependency/urgency):
1. #3 OMX Cross-Model Transition (urgent, independent)
2. #1 Superpowers Discipline Embedding + Model Upgrade (independent)
3. #4 Session-End Preparation System (depends on Step 12)
4. #5 Parallel Session Native Integration (largest scope)
5. #2 Visual Brainstorming (depends on Superpowers server)

## Feature #1: Superpowers Discipline Embedding

### Problem
BMB agents lack concrete verification checklists and red-flag patterns. Superpowers has battle-tested discipline skills (verification-before-completion, systematic-debugging, TDD) but injecting full skills wastes context.

### Design

**Approach**: Cherry-pick core checklists and red flags from Superpowers skills, embed directly in agent prompt markdown files. Max 300 tokens per agent.

**Agent-Discipline Mapping**:

| Agent | Disciplines | Token Budget |
|-------|------------|-------------|
| Executor | Verification Gate (5-step) + Systematic Debugging (3-fix limit) | ~300 |
| Frontend | Verification Gate + Systematic Debugging | ~300 |
| Tester | TDD Red-Green checklist + Testing Anti-patterns (top 4) | ~250 |
| Verifier | Verification Gate (strengthen existing) | ~150 |
| Architect | YAGNI principle + Scope Check (multi-subsystem split) | ~150 |
| Simplifier | Minimal Viable Change principle | ~100 |
| Analyst | (none — retrospective, not implementation) | 0 |
| Writer | (none — docs only) | 0 |
| Consultant | (none — advisory role) | 0 |

**Insertion Format** (appended to each agent .md):

```markdown
## Discipline Rules

### Verification Gate
Before ANY completion claim:
1. IDENTIFY: What command proves this?
2. RUN: Execute it (fresh, not cached)
3. READ: Full output, check exit code
4. VERIFY: Output confirms claim?
5. ONLY THEN: Report completion

RED FLAGS — STOP if you catch yourself:
- Using "should work", "probably passes"
- Expressing satisfaction before verification
- Trusting previous run results

### Debugging Discipline
- Phase 1 (investigation) MUST complete before ANY fix attempt
- 3 failed fixes → STOP, report architectural concern to Lead
- One variable at a time, never bundled fixes
```

## Feature #2: Visual Brainstorming

### Problem
BMB brainstorming (Step 2) is text-only. Superpowers v5.0 introduced a browser-based visual companion with WebSocket server that significantly improves design discussions with mockups, diagrams, and comparisons.

### Design

**Approach**: Reuse Superpowers' brainstorm server directly (not fork). BMB's Lead agent calls start-server.sh during Step 2 when visual questions are anticipated.

**Integration Flow**:
```
Step 2 (Brainstorm) extension:
├── Visual questions anticipated?
│   ├── YES → start-server.sh
│   │   ├── SCREEN_DIR = .bmb/brainstorm-screens/{session_id}/
│   │   ├── Show browser URL to user
│   │   └── Write HTML files for visualization
│   └── NO → existing terminal conversation
│
├── Per-question: browser vs terminal
│   ├── Mockups, layouts, comparisons → browser
│   └── Concepts, requirements, text → terminal
│
└── After Step 3 approval → stop-server.sh
```

**Server Path Resolution**:
```bash
SUPERPOWERS_SCRIPTS="$HOME/.claude/plugins/cache/superpowers-dev/superpowers/*/skills/brainstorming/scripts"
```

**Content Types for BMB**:
- Architecture diagrams (component relationships)
- Trade-off comparison matrices
- UI mockups (when frontend scope)
- Data flow diagrams
- Before/after comparisons

**Differences from Superpowers**:
| Aspect | Superpowers | BMB |
|--------|-------------|-----|
| Agent | Single brainstorm agent | Lead (in-process) |
| Content | Mockups, diagrams | + architecture, trade-off matrices |
| Lifecycle | Manual start/stop | Auto-start at Step 2, auto-stop after Step 3 |
| Screen dir | `.superpowers/brainstorm/` | `.bmb/brainstorm-screens/` |

### Files Changed
- `skills/bmb/bmb.md` — Step 2 extension with visual companion gate
- `.bmb/` — add `brainstorm-screens/` to gitignore pattern

## Feature #3: OMX Cross-Model Transition

### Problem
`cross-model-run.sh` uses raw `codex exec` → 100% timeout rate (4/4 sessions). Root cause: MCP server loading overhead + unstable process management.

### Design

**Approach**: Replace `codex exec` with `omx exec` in cross-model-run.sh.

**Key Changes**:
1. `omx cleanup` before each cross-model invocation (clear zombie processes from previous sessions)
2. `codex exec $MCP_DISABLE_ARGS (dynamically disables all configured MCP servers)` to disable all MCP servers (cross-model agents don't need them — this eliminates the primary timeout cause)
3. Shell `timeout` remains for timeout management (codex exec has no native timeout flag)
4. `omx cleanup` is best-effort — if omx not installed, skip silently

**Root Cause Analysis**:
The 100% timeout rate (4/4 sessions) was caused by codex loading 8 MCP servers including a failing supermemory server. MCP initialization blocked before the prompt could even execute. Disabling MCP servers via `$MCP_DISABLE_ARGS (dynamically disables all configured MCP servers)` eliminates this overhead entirely.

**Script Changes**:
```bash
# Before (MCP 8개 로드 시도 → 블로킹 → 타임아웃)
timeout "$TIMEOUT" codex exec $MODEL_ARGS --full-auto -C "$WORKDIR" "$FULL_PROMPT"

# After (MCP 0개 + 좀비 정리 → 즉시 실행)
omx cleanup 2>/dev/null || true  # best-effort zombie cleanup
timeout "$TIMEOUT" codex exec $MODEL_ARGS $MCP_DISABLE_ARGS (dynamically disables all configured MCP servers) --full-auto -C "$WORKDIR" "$FULL_PROMPT"
```

**Failure Handling**:
- Timeout → immediate Claude-only fallback (no retry, per learnings #10)
- omx not found → skip cleanup, proceed normally
- omx cleanup failure → log warning, proceed anyway

### Files Changed
- `bmb-system/scripts/cross-model-run.sh` — replace codex exec with omx exec
- `.bmb/config.json` — add `cross_model.use_omx: true` option

## Feature #4: Session-End Preparation System

### Problem
When a BMB session ends, there's no structured handover for the next session. Users must remember context and manually formulate the next prompt.

### Design

**Approach**: Extend Step 12 with automatic next-session plan generation and one-line prompt, with user confirmation gate.

**Flow**:
```
Step 12: Cleanup + Session Prep
├── Existing: commit, push, index
└── New: Session Handover
    ├── 1) Generate next-session-plan.md
    │   ├── This session completed items
    │   ├── Incomplete/discovered follow-up tasks
    │   ├── Recommended recipe/scope for next session
    │   └── Warnings (failures, risks)
    │
    ├── 2) Generate one-line prompt
    │   └── "BMB: [specific task description]"
    │
    └── 3) User confirmation (AskUserQuestion)
        ├── YES → finalize plan, display prompt
        ├── NO → delete plan, end session
        └── Custom input → revise and regenerate
```

**Output File**: `.bmb/next-session-plan.md`

```markdown
# Next Session Plan
Generated: 2026-03-19 12:30 KST

## Completed This Session
- [x] Feature X implemented and verified
- [x] Tests passing (14/14)

## Next Steps
1. [P1] Fix edge case in auth flow (discovered during testing)
2. [P2] Add error boundary to dashboard component

## Recommended Start
Recipe: standard | Scope: auth-edge-case-fix

## One-Line Prompt
> BMB: auth flow edge case 수정 — 테스트 중 발견된 토큰 만료 시 무한 리다이렉트 이슈 해결해줘

## Notes
- cross-model verification passed on 2nd attempt
- simplifier removed 23 lines of dead code
```

### Files Changed
- `skills/bmb/bmb.md` — extend Step 12 with session handover logic
- `.bmb/` — add `next-session-plan.md` to tracked outputs

## Feature #5: Parallel Session Native Integration

### Problem
Concurrent BMB sessions sharing `.bmb/` cause handoff file collisions (learnings #9). Current workaround (staging docs) is manual and error-prone.

### Design

**Core Concept**: SESSION_MODE enum with 3 values.

```
SESSION_MODE
├── standalone (default, 100% backward compatible)
├── sub (parallel track worker — code only, no docs/ edits)
└── consolidation (merge only — no code, docs merge only)
```

### 5.1 Track Splitting (Step 2 Extension)

```
Step 2 (Brainstorm) extension:
├── Existing brainstorming
└── New: Parallel feasibility check
    └── "Can this work be split into independent tracks?"
        ├── NO → standalone (existing behavior)
        └── YES → auto-generate:
            ├── .bmb/parallel-manifest.json
            ├── Track prompts (N per track)
            └── Consolidation prompt (1)
```

**parallel-manifest.json**:
```json
{
  "session_id": "main-2026-03-19",
  "tracks": [
    {
      "id": "auth",
      "scope": "auth flow edge case fix",
      "prompt": "BMB sub: auth flow 토큰 만료 무한 리다이렉트 수정",
      "worktree": ".bmb/worktrees/auth",
      "status": "pending"
    },
    {
      "id": "dashboard",
      "scope": "dashboard error boundary",
      "prompt": "BMB sub: dashboard 컴포넌트 에러 바운더리 추가",
      "worktree": ".bmb/worktrees/dashboard",
      "status": "pending"
    }
  ],
  "consolidation": {
    "prompt": "BMB consolidate: auth + dashboard 트랙 취합",
    "depends_on": ["auth", "dashboard"]
  }
}
```

### 5.2 Sub Mode Pipeline

```
Sub mode (reduced pipeline):
Step 1: Setup (SESSION_MODE=sub, read manifest)
Step 2: Skip (brainstorming already done)
Step 3: Skip (already approved)
Step 4: Architecture (scoped to own track)
Step 5: Execution (own worktree only)
Step 5.5: Commit (NO merge — consolidation does that)
Step 6-7: Test + Verify (own scope only)
Step 8: Reconciliation (own track only)
Step 9: Simplify
Step 10: Writer → staging docs ONLY (docs/ direct edit FORBIDDEN)
Step 11-12: Session end + manifest status update
```

### 5.3 Consolidation Mode

```
Consolidation mode (lightest recipe):
Step 1: Setup (read manifest, verify all tracks complete)
Step 2-4: Skip
Step 5: Merge worktree changes
Step 5.5: Resolve conflicts
Step 6-7: Integration test + verify (full scope)
Step 8: Full reconciliation
Step 9: Skip (already done in sub)
Step 10: Writer → merge staging docs + final docs/ update
Step 11-12: Normal cleanup
```

### Constraints
- Sub mode: `docs/` direct edit **absolutely forbidden** (learnings #9)
- Consolidation: **no code writing** — merge + docs only
- Standalone: existing behavior **100% backward compatible**
- Each sub session uses its own worktree (no shared working directory)

### Files Changed
- `skills/bmb/bmb.md` — SESSION_MODE routing, Step 2 track splitting, sub/consolidation pipelines
- `skills/bmb/SKILL.md` — add SESSION_MODE documentation
- `.bmb/config.json` — add `session.mode` field
- New: consolidation recipe in recipes table

## Feature #6: Consultant Real-Time Sync + Monitor Watchdog

### Problem 1: Consultant Stale Context
Consultant only re-reads `consultant-feed.md` when a SendMessage arrives. If the user switches to the Consultant pane without sending a message, Consultant has stale context. tmux has no "pane focused" event hook.

### Problem 2: Orphaned/Crashed Panes
Monitor currently only watches registered watch items. Unregistered zombie panes, crashed panes, and unacknowledged stall reports go undetected.

### Design

**Approach**: Extend Monitor with two new capabilities: (1) consultant feed heartbeat and (2) tmux pane watchdog with Lead nudge escalation.

**Consultant Heartbeat Flow**:
```
Monitor (every 60s):
  └─ Check .bmb/consultant-feed.md mtime
  └─ Changed since last check?
      ├─ YES → SendMessage to Consultant:
      │   {"type":"feed_update",
      │    "new_lines": N,
      │    "latest_event": "Step 5 executor complete"}
      └─ NO → skip (no message)

Consultant (on receiving heartbeat):
  └─ Re-read .bmb/consultant-feed.md
  └─ Update internal context
  └─ Ready to respond immediately when user switches to pane
```

**Watchdog Flow**:
```
Monitor (every 60s):
  └─ tmux list-panes -s -F '#{pane_id} #{pane_pid} #{pane_dead}'
  └─ For each pane NOT matching Lead or Consultant pane IDs:
      ├─ pane_dead=1 → SendMessage to Lead:
      │   {"type":"watchdog","event":"pane_dead","pane":"ID","ts":"HH:MM"}
      └─ pane alive but no registered watch item → SendMessage to Lead:
          {"type":"watchdog","event":"untracked_pane","pane":"ID","pid":N,"ts":"HH:MM"}

Lead Nudge Escalation:
  └─ If stall/process_died report gets no Lead acknowledgment within 120s:
      └─ Re-send with escalation: {"type":"watchdog","event":"nudge_repeat",...}
      └─ Maximum 3 nudges per event (then stop to avoid noise)
```

**Monitor Changes**:
- Add `consultant_feed_mtime` tracking to observation loop
- New SendMessage target: Consultant (for feed heartbeat)
- Add tmux pane sweep every 60s (metadata only, no content reading)
- Add nudge escalation with 120s ack timeout and 3-nudge cap
- Track Lead/Consultant pane IDs (read from `.bmb/consultant-pane-id` + own parent pane)

**Consultant Changes**:
- Handle new message type `feed_update` from Monitor
- On receipt: re-read feed, update context silently (no user-facing output)

### Files Changed
- `agents/bmb-monitor.md` — add consultant heartbeat + watchdog mode + nudge escalation
- `agents/bmb-consultant.md` — add feed_update message handler
- `skills/bmb/bmb.md` — Monitor spawn includes Consultant as SendMessage target + watchdog event handling
